# 好看智能相框 — 照片权限交互规范

> 面向 Android / iOS 原生开发，描述照片图库权限申请、状态流转、各权限级别下的页面展示逻辑与用户引导策略。

---

## 一、背景与设计原则

好看智能相框 App 的核心功能依赖于读取用户照片图库。自 iOS 14 / Android 14 起，系统支持「有限照片访问」（Limited Photo Access），用户可仅授权部分照片而非全部。App 需针对三种权限级别提供差异化的 UI 展示与引导体验。

### 设计原则

- **尊重用户选择**：不强制要求全部权限，所有权限级别下均提供可用体验
- **渐进式引导**：通过场景化的提示帮助用户理解「更多权限 = 更好体验」，而非反复弹窗
- **系统一致性**：权限申请弹窗和照片选择器均为**系统级弹窗**，App 不自定义这两个界面
- **状态可逆**：用户可随时通过系统设置更改权限，App 实时响应变更

---

## 二、权限级别定义

| 权限级别 | 系统枚举 (iOS) | 含义 | App 可访问范围 |
|---|---|---|---|
| **完全访问** (full) | `.authorized` | 用户允许访问所有照片 | 照片图库全部内容 |
| **有限访问** (limited) | `.limited` | 用户仅授权部分照片 | 用户手动选择的照片子集（可能为 0 张） |
| **无权限** (none) | `.denied` / `.restricted` | 用户拒绝授权或受限制 | 无法访问任何照片 |
| **未决定** (unknown) | `.notDetermined` | 用户尚未做出选择 | 无（等待授权） |

> **重要边界**：`limited` 状态下，用户可能选择了 0 张照片。此时系统仍报告 `.limited`（而非 `.denied`），但 App 实际可访问照片数为 0。这与「无权限」是不同的状态，需要差异化处理（详见 4.2.1）。

## 三、启动流程与相框绑定（前置步骤）

在进行照片权限申请之前，App 需要先引导用户完成相框绑定。照片权限申请发生在绑定成功之后。

### 3.1 启动流程概览

```
App 启动
  └─ 检查 FRAME_DEVICES 是否为空
       ├─ 无设备 → 展示欢迎引导页
       │     └─ 用户点击「扫码绑定相框」
       │           └─ 打开模拟扫码页面
       │                 └─ 扫描动画 2.5s → 识别成功
       │                       └─ 弹出「发现设备」确认弹窗
       │                             ├─ 确认绑定 → 关闭扫码页 → Toast 提示 → 触发照片权限申请
       │                             └─ 取消 → 继续扫描
       └─ 有设备 → 直接检查照片权限
             ├─ 未决定 → 照片权限弹窗
             └─ 已决定 → 进入主页
```

### 3.2 欢迎引导页

当用户首次打开 App 且未绑定任何相框时，展示全屏欢迎页。

**页面内容**：
- App 图标（渐变蓝色圆角矩形 + 相框图标）
- 标题：「绑定你的智能相框」
- 副标题：「扫描相框上的二维码完成绑定，随时将手机中的精彩照片发送到相框。」
- 示意插图：相框 + 手机扫码示意
- 主按钮：「扫码绑定相框」

> **设计要点**：用户必须先完成绑定才能使用 App，不提供「跳过」选项。

### 3.3 模拟扫码页面

全屏深色扫码界面，模拟相机取景效果。

**页面元素**：
- 顶部栏：「取消」按钮 + 标题「扫描二维码」
- 中央取景框：250×250px，四角白色 L 型标记线
- 取景框内：蓝色渐变扫描线动画（2s 循环）
- 取景框下方：提示文字「将相框上的二维码放入框内」
- 底部：手电筒按钮（装饰性）

**扫码流程**：
1. 页面打开后扫描线动画自动开始
2. 2.5 秒后自动识别成功
3. 取景框出现绿色勾号动画 + 震动反馈
4. 0.6 秒后弹出「发现设备」确认弹窗

### 3.4 发现设备确认弹窗

采用 iOS Alert 风格弹窗：
- 蓝色渐变相框图标
- 标题：「发现相框设备」
- 描述：展示设备名称和位置，如「好看智能相框 · 客厅」
- 主操作：「绑定此相框」→ 执行绑定并进入照片授权
- 取消操作：「取消」→ 关回扫码页继续扫描

### 3.5 绑定成功

绑定确认后：
1. 关闭扫码页面和欢迎引导页
2. 展示 Toast 提示：「绑定成功 · 好看智能相框 (客厅)」
3. 渲染相框设备列表
4. 延迟 1.2 秒后触发照片权限申请

### 3.6 后续添加设备

已绑定设备后，用户仍可通过以下入口继续添加更多设备：
- 首页 Header 右上角扫码按钮
- 「相框」Tab 页的「添加设备」按钮

## 四、权限申请流程

### 4.1 首次触发时机

照片权限申请在**首台相框绑定成功后**自动触发（延迟 1.2 秒），而非 App 启动时立即弹出。

> **技术说明**：iOS 调用 `PHPhotoLibrary.requestAuthorization(for: .readWrite)`，系统自动弹出权限弹窗。Android 调用 `READ_MEDIA_IMAGES` 权限请求。

### 4.2 系统权限弹窗（系统 UI，非 App 自定义）

弹窗由操作系统渲染，App 无法自定义样式和布局，仅可通过 `Info.plist` 中的 `NSPhotoLibraryUsageDescription` 设置描述文案。

**iOS 系统弹窗内容**：

```
┌─────────────────────────────┐
│    [App 图标]                │
│                             │
│  "好看智能相框"想访问您的照片  │
│                             │
│  允许访问照片图库以将精选照片  │
│  发送到智能相框，App 会智能   │
│  分析您的照片以推荐精选内容。  │
│                             │
├─────────────────────────────┤
│        选择照片…             │ ← 进入系统照片选择器
├─────────────────────────────┤
│    允许访问所有照片（粗体）    │ ← 授予完全访问权限
├─────────────────────────────┤
│        不允许（灰色）         │ ← 拒绝授权
└─────────────────────────────┘
```

**三个系统按钮的行为**：

| 按钮 | 行为 | 回调结果 |
|---|---|---|
| **选择照片…** | 系统弹出照片选择器（见 4.3），用户手动勾选允许访问的照片 | 权限状态 → `limited` |
| **允许访问所有照片** | 直接授予完全访问权限 | 权限状态 → `full` |
| **不允许** | 拒绝授权，弹窗关闭 | 权限状态 → `none` |

### 4.3 系统照片选择器（系统 UI，非 App 自定义）

当用户在权限弹窗中点击「选择照片…」，或后续通过系统设置修改为「选择的照片」时，系统会弹出原生的照片选择器。

**系统照片选择器特征**：

- 由系统直接渲染，App 无法控制其 UI 样式
- 展示用户设备中的所有照片（按时间倒序排列）
- 支持多选，用户逐张勾选要授权的照片
- 顶部有「取消」和「完成」按钮
- 完成后，系统将选中照片的访问权限授予 App

> **重要**：App 在收到 `limited` 权限回调后，只能通过 `PHPhotoLibrary.shared().presentLimitedLibraryPicker(from:)` 触发系统选择器让用户追加授权照片，不能自行实现选择 UI。

## 五、各权限级别下的页面展示

### 5.1 完全访问（full）— 正常体验

页面正常加载所有内容，无任何权限相关提示。

**展示内容**：
- ✅ 精选推荐区（AI 分析全部照片生成月度精选）
- ✅ 人物识别区（基于全部照片识别人脸并聚合）
- ✅ 地点识别区（基于全部照片的地理信息聚合）
- ✅ 所有待发送照片网格（按年/月/日分组，支持时间轴滚动）

### 5.2 有限访问（limited）

有限访问存在两个子状态，需要差异化处理。

---

#### 5.2.1 有限访问 + 已授权 0 张照片 — 空选状态

用户在系统权限弹窗中点击了「选择照片…」，系统弹出照片选择器，但用户**未选择任何照片就点了完成**。此时系统权限状态为 `.limited`，但 App 实际可访问照片数量为 0。

> **注意**：此状态与「无权限 / none」有本质区别 —— 用户**并未拒绝授权**，只是还没选择具体照片。引导策略应更积极地鼓励用户选择照片，而非引导去设置页。

**展示内容**：
- ⚠️ **有限访问提示横幅**（描述文案变为「尚未选择任何照片 · 请选择要授权的照片」）
- ✅ **空选引导页**（居中展示）
- ❌ 精选推荐区 — 隐藏（无照片可分析）
- ❌ 人物识别区 — 隐藏
- ❌ 地点识别区 — 隐藏
- ❌ 照片网格 — 隐藏

**空选引导页**：

```
         ┌──────────┐
         │   🖼️     │  ← 橙色调图标（区别于无权限的灰色）
         └──────────┘
      尚未选择任何照片

  您已允许部分照片访问，但还没有
  选择要授权的照片。请选择要发送
  到相框的照片。

      ┌──────────────┐
      │  🖼️ 选择照片  │
      └──────────────┘
```

| 元素 | 规格 | 与「无权限」空状态的差异 |
|---|---|---|
| 图标容器 | 80×80px，圆角 24px，背景 `rgba(255,149,0,0.08)` | 橙色调（无权限为灰色 `#F3F4F6`） |
| 图标 | 照片图标，36px，颜色 `#FF9500` | 橙色（无权限为灰色 `#9ca3af`） |
| 标题 | 「尚未选择任何照片」，18px Bold | 文案不同于「无法访问照片」 |
| 描述 | 13px，`rgba(60,60,67,0.6)`，行高 1.5 | 积极鼓励选择，而非说「请允许访问」 |
| 按钮 | 「选择照片」，15px SemiBold，白字蓝底 | 调起系统照片选择器（而非跳转设置） |

**点击「选择照片」**：调用 `presentLimitedLibraryPicker` 弹出**系统照片选择器**，用户可勾选要授权的照片。

**横幅仍然可见**，用户也可点击横幅上的「管理授权」弹出操作菜单。

---

#### 5.2.2 有限访问 + 已授权 N 张照片（N ≥ 1）— 可用但受限

页面正常加载，但额外展示**有限访问提示横幅**引导用户扩大授权范围。

**展示内容**：
- ⚠️ **有限访问提示横幅**（固定在页面顶部，精选区上方）
- ✅ 精选推荐区（仅基于已授权照片分析，内容可能较少）
- ✅ 人物识别区（仅基于已授权照片识别，人物可能不全）
- ✅ 地点识别区（仅基于已授权照片，地点可能不全）
- ✅ 待发送照片网格（仅展示已授权照片中的待发送部分）

**有限访问提示横幅**：

```
┌────────────────────────────────────────┐
│  ⚠️  仅可访问部分照片            管理授权 │
│      已授权 N 张照片 · 可能遗漏精彩内容   │

```

| 元素 | 规格 |

| 容器 | 圆角 12px，背景 `rgba(255,149,0,0.06)`，边框 1px `rgba(255,149,0,0.12)` |

| 描述 | 「已授权 N 张照片 · 可能遗漏精彩内容」，11px，`rgba(60,60,67,0.6)` |

#### 点击「管理授权」→ 弹出操作菜单（App Action Sheet）

横幅上的「管理授权」按钮在 4.2.1 和 4.2.2 两种子状态下**均可点击**，弹出同一个操作菜单。

## 六、权限变更响应

### 6.1 系统设置中的权限选项

用户在「系统设置 → 好看智能相框 → 照片」中可看到以下选项：

```
允许"好看智能相框"访问

  ✓ 无
    选择的照片          N 张照片 >
    所有照片
```

| 选项 | 效果 |
|---|---|
| **无** | 撤回全部权限 → App 切换到空状态引导页 |
| **选择的照片** | 点击后弹出系统照片选择器，重新勾选授权照片 → App 切换到有限访问模式 |
| **所有照片** | 授予完全访问 → App 切换到正常模式 |

### 6.2 App 响应机制

| 平台 | 监听方式 | 时机 |
|---|---|---|
| iOS | `PHPhotoLibraryChangeObserver` | 用户从系统设置返回 App 时（`applicationDidBecomeActive`） |
| Android | `onResume` 中重新检查权限状态 | Activity 恢复前台时 |

**响应流程**：

1. App 回到前台时检查当前权限状态
2. 若权限状态与内存中缓存的不同，执行状态切换
3. 重新加载照片数据并刷新 UI
4. 清空当前选中状态（避免引用已失效的照片）

## 八、关键交互细节

### 8.1 系统弹窗 vs App 弹窗

| 弹窗 | 归属 | App 可控 |
|---|---|---|
| 权限申请弹窗（首次） | **系统** | 仅可设置描述文案（`NSPhotoLibraryUsageDescription`） |
| 照片选择器（选择授权范围） | **系统** | 不可控，由 `presentLimitedLibraryPicker` 调起 |
| 系统设置页 | **系统** | 不可控，由 `openSettings` 跳转 |
| 「管理授权」操作菜单 | **App** | 完全可控，App 自定义 Action Sheet |
| 有限访问提示横幅 | **App** | 完全可控，App 自定义 UI |
| 无权限空状态引导页 | **App** | 完全可控，App 自定义 UI |
| 有限访问空选引导页 | **App** | 完全可控，App 自定义 UI |

### 8.2 「无权限」与「有限访问但 0 张照片」的区别

这是两个**完全不同的状态**，对应不同的 UI 和引导策略：

| 维度 | 无权限 (none) | 有限访问 + 0 张 (limited, N=0) |
|---|---|---|
| **系统权限状态** | `.denied` | `.limited` |
| **用户意图** | 明确拒绝了授权 | 同意授权但尚未选择具体照片 |
| **引导方向** | 去系统设置开启权限 | 选择照片（无需改权限） |
| **空状态图标色** | 灰色 (`#9ca3af`) | 橙色 (`#FF9500`) |
| **空状态标题** | 无法访问照片 | 尚未选择任何照片 |
| **空状态描述** | 请允许…访问照片图库 | 您已允许部分照片访问，但还没有选择… |
| **主操作按钮** | 「前往设置」→ 跳转系统设置 | 「选择照片」→ 调起系统照片选择器 |
| **有限访问横幅** | 不显示 | 显示（文案为「尚未选择任何照片」） |
| **App 可直接解决** | ❌ 需要用户去系统设置 | ✅ 可直接调起选择器 |

### 8.3 首次弹窗的一次性

- 系统权限弹窗**仅在首次请求时弹出一次**
- 用户做出选择后，后续无法通过代码再次触发系统弹窗
- 若用户首次选择了「不允许」，后续只能引导用户去系统设置手动更改

### 8.4 有限访问模式下的「继续添加照片」

- 此操作调用 `presentLimitedLibraryPicker`，弹出的是**系统照片选择器**
- 选择器会展示用户设备中的所有照片，已授权的照片会标记为已选中状态
- 用户可在此基础上追加勾选更多照片，也可取消已选照片的授权
- 选择完成后，App 收到回调，重新获取可访问照片列表并刷新页面

### 8.5 AI 分析与权限的关系

| 权限级别 | 精选分析 | 人物识别 | 地点识别 |
|---|---|---|---|
| **full** | 全量照片分析，结果完整 | 全量照片识别，人物齐全 | 全量照片识别，地点齐全 |
| **limited (N ≥ 1)** | 仅分析已授权照片，结果可能不完整 | 仅识别已授权照片中的人脸 | 仅识别已授权照片中的地理信息 |
| **limited (N = 0)** | 不执行分析（隐藏区域） | 不执行识别（隐藏区域） | 不执行识别（隐藏区域） |
| **none** | 不执行分析 | 不执行识别 | 不执行识别 |

### 8.6 权限变更时的数据清理

权限从高级别降低到低级别时（如 full → limited 或 limited → none）：
- 清空当前照片选中状态
- 移除底部选择栏
- 重新渲染照片网格（仅保留仍有权限访问的照片）
- 如有正在进行的发送任务，任务不受影响（已读取的照片数据仍在内存中）

## 九、iOS / Android 平台差异

| 特性 | iOS | Android |
|---|---|---|
| 有限照片访问 | iOS 14+ 原生支持 `.limited` 状态 | Android 14+ 支持 `READ_MEDIA_VISUAL_USER_SELECTED` |
| 系统照片选择器 | `presentLimitedLibraryPicker` | `MediaStore` + 权限回调 |
| 权限描述文案 | `Info.plist` → `NSPhotoLibraryUsageDescription` | `AndroidManifest.xml` → 权限声明 |
| 跳转系统设置 | `UIApplication.shared.open(URL(string: UIApplication.openSettingsURLString)!)` | `Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS)` |
| 权限变更监听 | `PHPhotoLibraryChangeObserver` + `applicationDidBecomeActive` | `onResume` 重新检查 |

## 十、原型说明

在 HTML 原型 (`index.html`) 中，由于无法调用真实系统 API，以上系统级弹窗和照片选择器均通过**模拟 UI** 实现，以便演示完整交互流程：

| 真实行为 | 原型中的模拟方式 |
|---|---|
| 系统权限弹窗 | 模拟 iOS Alert 样式的自定义弹窗 |
| 系统照片选择器 | 模拟全屏照片网格选择页 |
| 跳转系统设置 | 模拟 iOS 设置页风格的权限切换页 |
| App Action Sheet | 模拟 iOS Action Sheet 样式 |

> **开发注意**：原型中的模拟 UI 仅供演示流程参考。实际开发中，「权限申请弹窗」和「照片选择器」**必须使用系统原生 API**，不可自行实现 UI。仅「管理授权操作菜单」、「有限访问提示横幅」、「空状态引导页」为 App 自定义 UI。
