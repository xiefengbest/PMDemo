# 智能精选流程分析

基于当前代码对「智能精选」从触发到落库的完整过程做梳理。

---

## 一、整体流程（单月）

```
识别完成（该月已识别照片就绪）
    ↓
onSmartSelectStart(monthKey)  →  UI 标记「精选中」
    ↓
【快速精选】runQuickSelectForMonth
    - 时间分组，每组取第一张
    - 将「未取到」的照片 markPhotosAsSkipUpload
    - 返回 (count, coverUri)
    ↓
onSmartSelectDone(monthKey, count, coverUri)  →  UI 用回调更新封面与张数
    ↓
【清空】clearSkipUploadForMonth  →  该月所有 skipUpload = 0
    ↓
【全量精选】runSmartSelectForMonth
    - SmartSelectPipeline.process(metadataList)
    - 将 mergedPhotoIds markPhotosAsSkipUpload
    ↓
onSmartSelectDone(monthKey, null, null)  →  ViewModel 从 DB 加载封面与张数
```

- **快速精选**：只为尽快出封面和张数，规则简单（每组取第一张）。
- **全量精选**：在清空 skipUpload 后重跑完整管道（时间分组 + dHash 去重 + 质量选择），结果写回 skipUpload。

---

## 二、全量精选管道（SmartSelectPipeline.process）

**输入**：该月「已识别且有效」的照片（排除截图/文档/文字/二维码），`List<PhotoMetadata>`。

**步骤**：

| 步骤 | 说明 | 组件 |
|------|------|------|
| 1 | 按 `getSortTime()` 排序，二级排序 `generateId()` 保证稳定 | 内联 |
| 2 | 时间窗口分组：≤5s 且同一天为一组；无 dateTaken 时窗口 500ms | `TimeGrouping.groupByBurstTime` |
| 3 | 对每个连拍组 **并行** 调用 `processGroup(group)` | `async` + `awaitAll` |
| 4 | 汇总各组的 selectedPhotos、mergedPhotoIds、finalScores，**按综合分降序**再按时间+id 排序 | 内联 |

**输出**：`SmartSelectResult(selectedPhotos, mergedPhotoIds, groupStats)`。

---

## 三、单组处理（processGroup）

每个 **BurstGroup** 内：

1. **单张**：直接返回 `(listOf(photo), emptyList(), listOf(0.5f))`。
2. **多张**：
   - **dHash**：对组内每张图生成灰度缩略图 → `DHashCalculator.calculate` 得到 64-bit hash。
   - **相似簇（并查集）**：两两用 `SimilarityChecker.isSimilar(hash1, hash2)`（汉明距离 ≤12）判断，相似则 union，得到多个簇。
   - **簇内代表**：每个簇若仅 1 张则直接为代表；多张则用 `bestPhotoSelector.selectBestPhoto(clusterGroup)` 选**综合分最高的一张**作为代表。
   - **similarPhotoIds**：组内所有**非代表**的 id（相似去重阶段被合并掉的）。
   - **质量选择**：对「各簇代表」组成的 `deduplicatedGroup` 调用 `bestPhotoSelector.selectBestPhotos(deduplicatedGroup)`（Tenengrad + finalScore 门槛），得到本组最终保留与质量合并 id。
   - **合并结果**：本组 `mergedPhotoIds = similarPhotoIds + selectionResult.mergedPhotoIds`（相似重复 + 质量未达标）。

---

## 四、时间分组（TimeGrouping）

- **BURST_TIME_WINDOW_MS = 5000**：有 dateTaken 时，时间差 ≤5 秒且同一天归为一组。
- **DATE_ADDED_WINDOW_MS = 500**：双方都无 dateTaken 时用 500ms，避免批量导入被误合成一大组。
- 排序：`compareBy({ getSortTime() }, { generateId() })`。

---

## 五、相似度（SimilarityChecker）

- **汉明距离**：`xor` 后 `Long.bitCount`。
- **判定**：`distance <= 12`（HAMMING_THRESHOLD）视为相似。组内通过**并查集聚类**得到多个簇，簇内用 `selectBestPhoto` 选综合分最高的一张作为代表，其余进入 similarPhotoIds。

---

## 六、组内质量选择（BestPhotoSelector）

**selectBestPhoto**（簇内选代表）：综合分（finalScore）最高的一张，用于相似簇代表选择。

**selectBestPhotos**（对簇代表做质量筛选）：

1. **评分**：对组内每张照片并行算 `calculatePhotoScore`：
   - 灰度缩略图 → Laplacian 清晰度 + **Tenengrad**（SharpnessCalculator.calculateFull）；
   - 原图人脸检测（FaceDetectorWrapper）；
   - baseScore = 归一化清晰度×0.7 + (有人脸 ? 1.0 : 0.0)×0.3；
   - **finalScore** = base×WB + normLap×WL + normTen×WT + absLap×WAL + absTen×WAT（组内相对 + 绝对清晰度；小组时 normLap/normTen 压缩到 [0.5, 0.7]）。
2. **合格条件**（需同时满足）：
   - **Tenengrad ≥ TENENGRAD_FLOOR(900)**，否则视为模糊/假清晰可丢弃；
   - **finalScore > 门槛**：小组（≤2 张）用 FINAL_SCORE_FLOOR_SMALL_GROUP(0.25)，否则 FINAL_SCORE_FLOOR(0.4)。
3. **保留规则**：
   - 有「Tenengrad 达标且 finalScore 达标」的：这些**全部保留**，按 finalScore 降序、时间、id 排序；
   - 仅有 Tenengrad 达标、无 finalScore 达标的：只保留综合分最高的 **1 张**；
   - 组内无人 Tenengrad 达标：只保留综合分最高的 **1 张**。
4. **mergedPhotoIds**：组内未保留的 id（质量未达标或 Tenengrad 未达标被合并的）。

---

## 七、配置参数（SmartSelectConfig）

| 参数 | 值 | 说明 |
|------|---|------|
| NORMALIZED_SHARPNESS_FLOOR | 0.2 | 组内 Laplacian 归一化底边 |
| NORMALIZED_LAP_TEN_FLOOR | 0.2 | normLap/normTen 底边（常规组） |
| TENENGRAD_FLOOR | 900 | Tenengrad 阈值，低于视为模糊/假清晰 |
| FINAL_SCORE_FLOOR | 0.4 | 综合分门槛，高于才视为精选（常规组） |
| SMALL_GROUP_THRESHOLD | 2 | 组内张数 ≤ 此值视为小组 |
| NORMALIZED_LAP_TEN_FLOOR_SMALL_GROUP | 0.5 | 小组时 normLap/normTen 底边 |
| NORMALIZED_LAP_TEN_CEIL_SMALL_GROUP | 0.7 | 小组时 normLap/normTen 顶边 |
| FINAL_SCORE_FLOOR_SMALL_GROUP | 0.25 | 小组时综合分门槛（降低过度过滤） |
| LAP_ABS_MAX / TEN_ABS_MAX | 3000 / 20000 | 绝对清晰度映射上限 |
| WEIGHT_BASE_SCORE 等 | 0.4, 0.2, 0.15, 0.15, 0.1 | finalScore 各分量权重 |

---

## 八、与 DB / UI 的衔接

- **写入**：`PhotoRepository.runSmartSelectForMonth` 中，`result.mergedPhotoIds` 分批调用 `photoDao.markPhotosAsSkipUpload(batch)`，即被合并/淘汰的照片 `skipUpload = 1`。
- **读取**：`getSmartSelectedPhotosByMonth(monthKey)` 查询「已识别且 (skipUpload=0 或 NULL) 且未删除未发布」的照片，即精选结果列表。
- **UI**：首页月份卡片、月份详情列表、封面与张数，在「精选开关」打开时都基于上述精选结果（skipUpload=0）；发送到相框时使用的也是同一列表，故**精选会直接影响可发送到相框的图片集合**。

---

## 九、小结

- **两阶段**：快速精选（每组取一）→ 清空 skipUpload → 全量精选（dHash + 质量），全量结果才持久化。
- **全量管道**：时间分组（5s/500ms）→ 组内 dHash + **并查集聚类**（汉明≤12），簇内 **selectBestPhoto** 选 1 张代表 → 对代表做 **Tenengrad + finalScore** 质量筛选（合格多张全留，否则留最高 1 张）→ 未保留的 id 写入 skipUpload=1。
- **最终列表**：按**综合分降序**，再按时间、id 排序，保证质量优先且顺序稳定。
- **稳定性**：排序与比较均带 `generateId()` 二级排序，相同输入多次运行结果一致。
