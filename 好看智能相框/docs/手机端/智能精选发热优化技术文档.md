# 智能精选发热优化技术文档

针对大批量照片（如 1700+ 张）进行精选时手机发热严重的问题，本方案通过限制精选管道并发与识别阶段节流，在发热与效率之间取得平衡。

---

## 一、问题背景

### 1.1 发热原因

| 阶段 | 位置 | 主要操作 | 发热原因 |
|------|------|----------|----------|
| **识别** | PhotoRecognitionManager | ML 分类（截图/文档/文字/二维码） | 连续 batch 推理，CPU/GPU 持续高负载 |
| **全量精选** | SmartSelectPipeline | dHash + Tenengrad + Laplacian + 人脸检测 | 每张图解码、灰度、卷积 |
| **并行** | 各组 processGroup | async + awaitAll | 所有连拍组同时执行，打满 CPU |

### 1.2 优化目标

- 降低 CPU 持续高负载，缓解发热
- 保持合理处理效率，避免耗时明显增加

---

## 二、优化方案

### 2.1 方案一：精选管道组间并发限制

**文件**：`feature/mobile/.../smartselect/SmartSelectPipeline.kt`

**原理**：原先所有连拍组通过 `async { processGroup(group) }.awaitAll()` 并行执行，组数多时 CPU 占用极高。通过 Semaphore 限制同时处理的组数，降低峰值负载。

**实现**：

```kotlin
// 根据 CPU 核心数动态计算许可数
val cores = Runtime.getRuntime().availableProcessors()
val permits = maxOf(2, minOf(cores - 1, 4))
val semaphore = Semaphore(permits)

val groupResults = timeGroups.map { group ->
    async {
        semaphore.withPermit { processGroup(group) }
    }
}.awaitAll()
```

**许可数公式**：`max(2, min(cores - 1, 4))`

- 至少 2：保证一定并行度
- 最多 4：控制发热
- `cores - 1`：预留 1 核给 UI 等

**典型设备效果**：

| CPU 核心 | 许可数 | 说明 |
|----------|--------|------|
| 4 核 | 3 | 中端机 |
| 6 核 | 4 | 高端机 |
| 8 核 | 4 | 上限 4，控制发热 |

**依赖**：`kotlinx.coroutines.sync.Semaphore`、`withPermit`

---

### 2.2 方案二：识别阶段批次间节流

**文件**：`feature/mobile/.../manager/PhotoRecognitionManager.kt`

**原理**：识别按 batch（30 张）连续执行，批次间无间隔。每处理 2 批后增加一次短 delay，降低持续高负载。

**实现**：

```kotlin
photosToRecognize.chunked(batchSize).forEachIndexed { batchIndex, batch ->
    // ... 识别、保存、更新进度 ...

    // 每 2 批休息 50ms，缓解持续高负载与发热，兼顾效率
    if (batchIndex > 0 && batchIndex % 2 == 0) {
        delay(50)
    }
}
```

**参数**：

- 节流间隔：每 2 批（即每 60 张）一次 delay
- delay 时长：50ms

**耗时影响估算**：

| 照片数 | 批次数 | delay 次数 | 额外耗时 |
|--------|--------|------------|----------|
| 300 | 10 | 4 | ~0.2s |
| 500 | 17 | 8 | ~0.4s |
| 1700 | 57 | 28 | ~1.4s |

## 三、效果与权衡

### 3.1 效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| CPU 峰值 | 多核打满 | 并发受限，峰值下降 |
| 持续负载 | 识别/精选连续高负载 | 有间隔，周期性降低 |
| 发热 | 明显 | 缓解 |

### 3.2 效率影响

| 方案 | 对耗时的影响 |
|------|--------------|
| Semaphore | 精选时间略增（组数多时更明显） |
| 识别 delay | 识别时间增加约 0.2s～1.5s（视照片数量） |

整体为「发热与效率折中」策略。

---

## 四、可选调优

### 4.1 偏重效率

- Semaphore 许可数：`max(2, min(cores - 1, 6))`
- 识别 delay：取消或改为每 3 批 delay(30)

### 4.2 偏重控温

- Semaphore 许可数：`max(2, cores / 2)`
- 识别 delay：每批 delay(80)

### 4.3 结合内存

```kotlin
val memoryClass = (context.getSystemService(ACTIVITY_SERVICE) as ActivityManager).memoryClass
val basePermits = maxOf(2, minOf(cores - 1, 4))
val permits = if (memoryClass < 256) maxOf(2, basePermits - 1) else basePermits
```
