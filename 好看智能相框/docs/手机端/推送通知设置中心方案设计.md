# 推送通知设置中心方案设计

## 一、背景与目标

为将对用户的打扰降到最低并充分尊重用户隐私，我们在客户端（Mobile）提供独立的「推送通知」页面（路径：**首页 - 我的 - 推送通知**），用于统一管理「精选合辑推荐通知」「系统与活动通知」等开关。

**目标**：为用户提供统一的推送开关管理界面，让用户能够根据自身需求，精细化控制自己想要接收的通知类型。

---

## 二、推送通知页 UI/交互设计

入口：**首页 - 我的 - 推送通知**

### 1. 界面结构

```text
【系统通知权限状态】
当前状态：[已开启/未开启]   (若未开启，展示"去设置"按钮跳转系统设置页)
--------------------------------------------------
【精选合辑推荐通知】
💡 每天根据本地相册为您推荐优质的合辑，一键发给家人
  开关：[开启 / 关闭] (默认开启)

【系统与活动通知】
💡 系统升级、相框状态变更及官方活动等重要消息
  开关：[开启 / 关闭] (默认开启)
```

### 2. 交互说明
- **系统通知权限联动**：顶部需检测设备操作系统层面的 Push 权限。如果用户在系统设置里关闭了整个 App 的通知权限，这里应该给出明显的横幅提示，并引导去系统级开启。如果系统权限被关，下方的各项开关虽然可以拨动（或者置灰），但实际不会有任何弹窗。
- **状态即时保存**：用户拨动开关后，客户端应立即将状态异步同步至服务端，并更新本地缓存。

---

## 三、各业务线推送链路调整

为了配合此设置中心，各路推送业务在下发/展示前必须增加一层**状态校验**拦截机制。

### 1. 精选合辑推荐通知（本地触达）
该推送采用 **"WorkManager 本地定时任务为主 + 友盟在线透传为辅"** 的双引擎架构（详见 `local_album_push_design.md`），通知内容完全在客户端本地生成。
- **拦截逻辑节点**：客户端本地
- **改造点**：无论是 WorkManager 定时触发还是友盟在线透传触发，进入判定链路后的**第一步需要先校验本地 SharedPreferences / MMKV 中的 `push_album_recommend_enabled` 开关状态**。若用户已在「推送通知」页关闭该开关，则直接丢弃，停止后续的本地相册查询及通知弹出逻辑。

### 2. 系统消息等（视级别而定）
- 对于最高优先级的系统公告、合规性提示，建议无视开关强制推送（或不在前端暴露关闭开关）。
- 普通的营销活动则交由服务端判断 `push_system_enabled` 状态。

---

## 四、数据结构设计 (User Settings Table)

针对推送开关，需要在用户设置表中新增相关的字段支持：

| 字段名 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `user_id` | string | - | 用户唯一标识 |
| `push_album_recommend` | boolean | true | 开启本地精选合辑推荐通知 |
| `push_system_msg` | boolean | true | 开启系统及活动通知 |

**客户端与服务端同步机制**：
- 客户端在登录/冷启动时，调用 `GET /user/settings` 拉取当前状态并覆盖本地缓存。
- 用户在「推送通知」页拨动开关时，立即调用 `PUT /user/settings/push` 接口同步至服务端，同时修改本地状态。

---

## 五、埋点监控

除了各推送的抵达、点击率外，需新增关于开关状态的数据埋点，以便观察用户对各类推送的反感情绪：
1. **开关关闭事件**：当用户主动关闭某一项推送时进行打点。
2. **状态大盘统计**：每日对活跃用户的各项推送开启比例进行汇总分析，若某项推送的关闭率激增，说明该推送频率过高或文案体验差，需及时调整策略。
