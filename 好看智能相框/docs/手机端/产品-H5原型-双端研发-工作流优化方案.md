# 产品 → HTML 原型 → Android/iOS 研发 工作流优化方案

> 目标：**更快产出 HTML 高保真原型**，并让 **Android / iOS 能完美复刻** 原型效果。

---

## 一、现状与目标

| 现状 | 目标 |
|------|------|
| 每次在 Cursor 里从零描述需求再生成 HTML，耗时长、易遗漏规范 | 固定「规范 + 模板 + 话术」，减少重复描述，一次 prompt 即可产出符合规范的 HTML |
| 交付研发时只有文档 + 原型链接，研发需自行对照 | 交付物标准化：原型链接 + 设计 token 引用 + 交互/状态说明 + 与《UI视觉规格》的章节对应 |
| 双端复刻依赖研发自行理解原型 | HTML 与《UI设计规范》《UI视觉规格文档》强绑定，研发按文档还原即可 |

---

## 二、加速 HTML 原型产出（产品侧）

### 2.1 每次生成前必引用的规范（在 Cursor 里 @ 引用）

生成或修改 HTML 原型时，**务必在对话中引用**以下文件，避免 AI 自由发挥：

| 引用文件 | 作用 |
|----------|------|
| `mobile_ui_ai_rules.md` | 约束布局、字体、组件、输出格式，确保可落地 |
| `UI设计规范.md` | 颜色/字体/间距等 Design Token，HTML 必须用 CSS 变量或与规范一致 |
| `UI视觉规格文档.md` | 具体页面的尺寸、毛玻璃、安全区等，生成时即按规格来 |

**推荐用法**：在 Cursor 里输入需求时，先写：

```
请严格按以下规范生成/修改本项目的 HTML 原型：
@mobile_ui_ai_rules.md
@好看智能相框/docs/手机端/UI设计规范.md
@好看智能相框/docs/手机端/UI视觉规格文档.md
```

然后再写具体需求（例如：「在欢迎页增加一个「跳过」按钮，样式见规范主色按钮」）。

### 2.2 需求描述模板（复制即用，减少来回）

每次提「新页面 / 新模块」时，可按下面结构写，便于 Cursor 一次产出完整结果：

```
【页面/模块名称】
- 所属：照片 Tab / 相框 Tab / 商城 / 我的 / 独立流程（如欢迎页）
- 主要元素：（列表、表单、卡片、弹窗等）
- 交互要点：（点击跳转、弹窗、多状态等）
- 状态：Loading / Empty / Error / 正常（若需要请说明）
- 对应《UI视觉规格文档》章节：（若有，如「二十七、欢迎页」）
- 模拟数据：用假数据即可，字段名用英文（如 title, count, list[]）
```

示例：

```
【欢迎页顶部「跳过」】
- 所属：独立流程（绑定引导页）
- 主要元素：右上角文字按钮「跳过」，主色、无背景
- 交互要点：点击后跳转到首页（在原型里用 hash 或 alert 模拟即可）
- 对应《UI视觉规格文档》章节：二十七、欢迎页（绑定引导页）
- 无需额外状态
```

这样 AI 会同时遵循规范与视觉规格，减少返工。

### 2.3 复用现有原型（改而不是从零写）

- 新需求若是「在现有页面上加一块」「改一个组件」，直接说：
  - 「在 `index.html` 的 XXX 区域增加……，保持现有风格，沿用 `UI设计规范.md` 的 token。」
- 打开 `index.html` 并 @ 该文件，让 AI 在现有结构上修改，避免重复生成整页。

### 2.4 模拟数据约定（方便双端对照）

- 列表、标题、数量等用**有意义的假数据**，字段命名清晰（如 `photoList[]`、`frameName`、`count`）。
- 若某页有固定数据结构，可在需求里写清，例如：「列表项包含：缩略图、标题、时间、发送状态」。
- 这样研发在实现时可直接对照原型里的数据结构设计 Model / 接口。

## 三、让 Android / iOS 完美复刻（交付标准化）

### 3.1 交付物清单（每次需求必含）

| 交付项 | 说明 | 用途 |
|--------|------|------|
| **1. 原型链接/文件** | 当前版本的 HTML（如 `index.html` 或带日期的副本） | 研发可直接打开查看交互与视觉 |
| **2. 本需求涉及的《UI视觉规格文档》章节** | 例如「二十七、欢迎页」「二、底部 Tab 栏」 | 研发按章节查数值，避免猜 |
| **3. 本需求涉及的《UI设计规范》Token** | 若用到了特殊颜色/字号/间距，列出 token 名（如 `color-primary`、`space-8`） | 双端实现时直接对 token，保证一致 |
| **4. 交互与状态说明** | 点击反馈、Loading/Empty/Error、弹窗出现消失、系统 UI 说明（见《照片权限交互说明文档》） | 避免漏状态、漏交互 |
| **5. 与现有说明文档的对应** | 若需求落在「照片权限」「发送任务」等已有文档，注明章节 | 研发可联动阅读 |

### 3.2 「完美复刻」的约定（产品与研发共识）

- **视觉**：以《UI视觉规格文档》和《UI设计规范》为准；HTML 原型是「可交互的规格示意图」，数值以文档为准，若文档与原型不一致，以文档为准（建议产品在定稿时同步更新文档）。
- **交互与状态**：以产品提供的「交互与状态说明」+ 现有交互文档为准；原型中的动效时长、反馈方式等若在规范中有写（如 150–250ms），按规范实现。
- **系统 UI**：权限弹窗、照片选择器等以《照片权限交互说明文档》为准，原生必须用系统 API，不按原型里的模拟 UI 实现界面。

### 3.3 可选：从 HTML 反哺文档

- 若某次在原型里新增了组件或页面，而《UI视觉规格文档》尚未覆盖，建议**同步补充**对应章节（颜色、字号、间距、圆角、毛玻璃等），便于后续双端与后续需求一致引用。

## 四、工作流小结（谁在何时做什么）

| 角色 | 动作 |
|------|------|
| **产品** | 1）用「需求描述模板」写清需求；2）在 Cursor 中 @ 三项规范 + 现有 `index.html`（若为修改）；3）生成/迭代 HTML 原型；4）按「交付物清单」整理：原型 + 规格章节 + Token + 交互说明；5）交付研发并注明关联文档章节。 |
| **Android / iOS** | 1）打开原型体验交互；2）按《UI视觉规格文档》对应章节与《UI设计规范》Token 实现界面；3）按交互说明与现有文档实现状态与系统能力；4）差异以文档为准，必要时与产品确认。 |

---

**使用建议**：  
- 产品：把「2.2 需求描述模板」存成自己的常用片段，每次新需求复制后填空。  
- 研发：要求产品按「3.1 交付物清单」交付，自己只认《UI视觉规格文档》+《UI设计规范》+ 交互说明，减少对原型「猜」的部分。

这样即可**更快执行出 HTML 结果**，并让 **Android / iOS 有据可依、完美复刻**。
