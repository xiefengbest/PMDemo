# 精选合辑推荐 Push 方案

## 一、背景与目标

- **背景**：客户端（Mobile）会在本地识别用户的相册照片，并自动生成剔除了杂图（如截图、二维码）的"精选合辑"（名称通常基于时间跨度，如"26年3月"）。用户往往不会一次性将所有合辑发送到智能相框中，导致优质内容留在手机本地。
- **目标**：每天定时向用户推送一个**未发送过**的精选合辑，引导用户点击进入合辑详情页并发送至相框，从而**提高用户的发图效率和相框的内容活跃度**。

---

## 二、方案演进与选型分析

在确定最终方案前，我们评估了三种可能的架构，并逐一排除了前两种不可靠的方案。

### 方案 A：服务端静默透传唤醒 + 客户端本地判定 ❌

**思路**：服务端定时下发空透传消息唤醒 App，客户端被唤醒后查询本地数据库、组装文案、弹出本地通知。

**不可行原因**：项目推送基于友盟 + 华为/小米/OPPO/vivo 厂商通道。各厂商对透传/静默消息的支持情况如下：

| 厂商 | 透传/静默推送 | App 被杀后可达 | 说明 |
|------|-------------|--------------|------|
| 华为 HMS | ⚠️ 有限支持 | ❌ | 仅 App 在线可达；被杀后不拉起（高优先级特权需额外申请且有配额） |
| 小米 MiPush | ❌ 已停止 | ❌ | 2022 年 9 月起全面停止透传消息下发 |
| OPPO Push | ❌ 不支持 | ❌ | 官方文档明确仅支持通知栏消息 |
| vivo Push | ❌ 不支持 | ❌ | 官方文档明确仅支持通知类消息 |
| Apple APNs | ⚠️ 不保证 | ❌ | 支持静默推送但系统会节流；用户强退后不唤醒；低电量模式失效 |

**结论**：5 个平台中没有任何一个能在 App 被杀的情况下可靠投递静默/透传消息，而"用户长时间未打开 App"恰恰是最需要触达的场景。

### 方案 B：WorkManager 本地定时任务驱动 ❌

**思路**：使用 Android WorkManager / iOS BGAppRefreshTask 注册每日定时任务，由操作系统调度执行本地判定逻辑。

**不可行原因**：国产 ROM（华为 EMUI/HarmonyOS、小米 MIUI/HyperOS、OPPO ColorOS、vivo OriginOS）对后台任务的限制极为激进：
- 华为 PowerGenie 会主动杀死所有非白名单应用的后台进程，后台执行时间限制约 60 分钟
- 小米/OPPO 的电池优化会将 WorkManager 任务延迟数小时甚至完全不执行
- 用户强杀 App 后，WorkManager 在多数国产 ROM 上无法重新调度
- 需要引导用户手动关闭电池优化、开启自启动等，操作路径因厂商而异且转化率极低

**结论**：WorkManager 在国产 ROM 上的可靠性不足以作为核心推送触发机制。

### 方案 C：合辑元数据上云 + 服务端筛选 + 厂商通知消息下发 ✅（最终方案）

**思路**：客户端将识别出的精选合辑**元数据**（不含照片）同步到服务端，由服务端完成筛选和文案组装，通过**厂商通知消息通道**（所有厂商稳定支持）下发到用户设备。

**优势**：
- 厂商的"通知消息"是所有平台最稳定、到达率最高的通道
- 筛选逻辑集中在服务端，便于灰度、策略调整、A/B 测试
- 不依赖 App 进程存活，不受国产 ROM 后台限制影响

## 三、最终方案：合辑元数据上云 + 服务端推送

### 1. 架构核心思想

客户端负责**识别和上报**，服务端负责**筛选和触达**。照片内容始终留在本地，仅上传合辑的描述性元数据。

### 2. 全链路时序流转图

```text
【客户端上报】  App 启动/前台运行时，扫描本地精选合辑列表
                将合辑元数据（名称、数量、是否已发送等）同步至服务端
                │
【服务端定时】  每天定时（如 20:00）执行推荐任务
                │
【服务端筛选】  从用户的合辑元数据中筛选：
                ├─ 有无"未发送到相框"的合辑？ → 没有则跳过
                ├─ 过滤近 N 天已推荐过的合辑
                └─ 按优先级排序，选出最佳推荐合辑
                │
【服务端组装】  根据合辑名称和照片数量，从文案模板池中组装推送标题和内容
                │
【厂商通道下发】 通过友盟服务端 API 下发"通知消息"（非透传）
                Android: 走华为/小米/OPPO/vivo 厂商通知通道
                iOS: 走 APNs alert 通知
                │
【系统级展示】  由操作系统直接在通知栏展示，无需 App 进程存活
                │
【用户点击】    点击通知 → 打开 App → 路由到合辑详情页
                │
【回调上报】    客户端上报点击事件；若用户发送了合辑，更新元数据状态
```

### 3. 隐私边界说明

| 数据类型 | 是否上云 | 说明 |
|---------|---------|------|
| 照片文件/缩略图 | ❌ 绝不上云 | 照片始终保留在用户设备本地 |
| 合辑元数据 | ✅ 上云 | 仅包含合辑名称、照片数量、是否已发送等描述信息 |

上传的元数据**不包含任何照片内容**，仅是合辑的统计性描述。需在隐私政策中明确告知用户该数据的收集和用途。

## 四、详细策略与规则说明

### 1. 客户端合辑元数据上报

#### 上报数据结构

```json
{
  "albums": [
    {
      "album_id": "local_202603",
      "album_name": "26年3月",
      "photo_count": 28,
      "valid_photo_count": 25,
      "sent_to_frame": false,
      "sent_photo_count": 0,
      "created_at": "2026-03-31T10:00:00Z"
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `album_id` | string | 客户端本地生成的合辑唯一标识 |
| `album_name` | string | 合辑显示名称（如"26年3月"） |
| `photo_count` | int | 合辑内照片总数 |
| `valid_photo_count` | int | 去除已删除照片后的有效数量 |
| `sent_to_frame` | boolean | 是否已全量/大部分发送到相框 |
| `sent_photo_count` | int | 已发送到相框的照片数量 |
| `created_at` | string | 合辑生成时间 |

#### 上报时机
- **App 冷启动**时，扫描本地合辑列表，与上次上报做 diff，有变化则上报
- **用户发送合辑到相框后**，立即上报该合辑的最新状态（`sent_to_frame` / `sent_photo_count` 更新）
- **新合辑生成时**（后台识别完成后），追加上报

#### 上报接口

```
PUT /api/v1/user/album-metadata
```

采用**全量覆盖**策略，每次上报完整的合辑元数据列表，避免增量同步的复杂性和一致性问题。

### 2. 服务端筛选策略（推哪个？）

服务端定时任务（如每天 20:00）遍历目标用户，依次执行：

1. **过滤已发送**：剔除 `sent_to_frame = true` 或 `sent_photo_count / photo_count > 80%` 的合辑。
2. **过滤已推荐**：剔除过去 N 天内已通过此功能推荐过的合辑（服务端记录推荐历史）。
3. **过滤失效内容**：剔除 `valid_photo_count < 3` 的合辑。
4. **优先级排序**：
   - 最新月份/时间段的合辑优先
   - "去年今日"类纪念合辑特殊加权
5. **无合辑可推**：若所有合辑都被过滤，该用户当天不推送。

### 3. 推送文案设计（推什么？）

服务端根据选中合辑的元数据，从文案模板池中动态组装：

**策略 A：基于合辑"时间名称"的动态组装（高优先级）**
- "为你整理了【{album_name}】的精彩瞬间，快分享给家人看看吧！"
- "【{album_name}】的记忆已打包，有 {valid_photo_count} 张好照片可以发到相框啦~"
- "不想错过【{album_name}】的美好？一键将这组精选照片同步到相框！"
- "回顾【{album_name}】的点滴，这组无杂图的纯净相册送给你。"

**策略 B：通用兜底文案**
- "相框有点想你了，刚帮你清理出 {valid_photo_count} 张好照片，发去看看吧~"
- "新鲜出炉的近期生活回顾，没有截图和杂图，赶紧一键发送吧！"

服务端随机选择句式，避免用户每天收到相同文案产生"广告疲劳"。

### 4. 推送下发

通过友盟服务端 API 下发**通知消息**（非透传），Payload 中携带 `album_id` 用于点击跳转：

```json
{
  "type": "notification",
  "title": "精选合辑推荐",
  "text": "为你整理了【26年3月】的精彩瞬间，快分享给家人看看吧！",
  "extra": {
    "action": "album_recommend",
    "album_id": "local_202603"
  }
}
```

通知消息由**操作系统直接展示**在通知栏，不依赖 App 进程存活。

### 5. 频控与防扰机制

- **下发频率**：服务端每用户每天最多下发 **1 条**合辑推荐通知。
- **活跃免扰**：如果服务端检测到用户当天已经有大量发图行为（通过合辑状态变更判断），可策略性跳过推送。
- **厂商频控适配**：通知消息同样受各厂商消息分类限制（公信/私信），需按各厂商要求申请合适的消息分类（建议申请为"私信"类别以获得不限量额度）。

### 6. 客户端通知点击处理

用户点击通知后，客户端从 Intent/Payload 中提取 `album_id`，路由到合辑详情页：
- 如果本地合辑仍然存在且有效 → 正常展示详情页
- 如果合辑已被删除或已全量发送 → 展示兜底页面或 Toast 提示

### 7. 通知生命周期管理

- **防堆积清理**：每次 App 回到前台（冷启动/热启动）时，主动调用 `NotificationManager.cancel` / `removeDeliveredNotifications` 清空所有驻留的"合辑推荐"类通知，避免用户进 App 发完合辑后通知栏仍有过期推送。

## 五、数据结构设计

### 1. 服务端合辑元数据表 (album_metadata)

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | 用户唯一标识 |
| `album_id` | string | 合辑唯一标识 |
| `album_name` | string | 合辑显示名称 |
| `photo_count` | int | 照片总数 |
| `valid_photo_count` | int | 有效照片数 |
| `sent_to_frame` | boolean | 是否已发送到相框 |
| `sent_photo_count` | int | 已发送照片数 |
| `created_at` | timestamp | 合辑生成时间 |
| `updated_at` | timestamp | 最近一次上报更新时间 |

### 2. 推荐历史表 (album_recommend_history)

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | 用户唯一标识 |
| `album_id` | string | 被推荐的合辑 ID |
| `push_date` | date | 推送日期 |
| `clicked` | boolean | 用户是否点击了该推送 |
| `converted` | boolean | 用户是否最终发送了该合辑 |

## 七、数据埋点与效果评估

| 漏斗节点 | 埋点指标 | 说明 |
|---------|---------|------|
| 元数据上报 | 上报成功率 | 监测客户端上报的稳定性 |
| 服务端筛选 | 有效用户占比 | 有可推荐合辑的用户比例 |
| 推送下发 | 下发量 / 到达率 | 监测厂商通道到达率 |
| 通知曝光 | Push 曝光量 | 实际在手机上展示通知的次数 |
| 用户点击 | Push 点击率 (CTR) | 点击通知的比例 |
| 发送转化 | 发送转化率 (CVR) | 点击后成功发送合辑到相框的比例 |
