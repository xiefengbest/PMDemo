# 人物聚类技术文档

## 1. 功能概述

人物聚类功能对本地相册中的照片进行人脸检测、特征提取和相似度聚类，将同一人物的照片自动归组，在"待发送"页面的「人物」模块展示，支持点击进入人物详情查看所有相关照片。

**核心特性：**
- 基于 ML Kit 人脸检测 + MobileFaceNet TFLite 模型的人脸嵌入提取
- 贪心最近邻聚类 + 出簇验证两阶段算法
- 增量聚类：已处理的照片不重复扫描，新增照片优先归入已有簇
- 跨批次一致性：多次触发聚类时同一人物不会被拆分为多个簇
- 质量过滤：自动排除侧脸过大、墨镜遮挡、图像模糊、人脸过小的照片

## 3. 核心模块说明

### 3.1 FaceEmbeddingExtractor

**文件**：`photoanalysis/face/FaceEmbeddingExtractor.kt`

负责从单张照片中检测人脸、执行质量过滤，并调用 TFLite 模型提取人脸嵌入向量。

#### 3.1.1 使用的模型

| 项目 | 值 |
|---|---|
| 模型文件 | `assets/mobile_face_net.tflite`（约 5MB） |
| 输入尺寸 | 112×112 像素，RGB |
| 输出维度 | 192 维 L2 归一化嵌入向量 |
| 加载方式 | `AssetFileDescriptor` + `FileInputStream.channel.map` → `MappedByteBuffer`（TFLite 要求） |
| 推理线程数 | 2 |

#### 3.1.2 质量过滤阈值

| 常量 | 值 | 过滤条件 | 说明 |
|---|---|---|---|
| `MAX_YAW_ANGLE` | 42° | `\|yaw\| > 42°` | 侧脸偏转过大，嵌入可靠性下降 |
| `MAX_PITCH_ANGLE` | 38° | `\|pitch\| > 38°` | 仰俯角过大 |
| `MIN_FACE_AREA_RATIO` | 0.002 | `faceArea/imageArea < 0.002` | 人脸过小（远景群体照中的背景人物） |
| `SUNGLASSES_EYE_PROB` | 0.70 | `leftEye < 0.70 AND rightEye < 0.70` | 双眼均被遮挡（墨镜、遮挡物）；侧脸单眼不可见不受影响 |
| `MIN_EYE_OPEN_PROB` | 0.05 | `leftEye < 0.05 AND rightEye < 0.05` | 双眼完全闭合（兜底，通常被上一条覆盖） |
| `BLUR_THRESHOLD` | 30.0 | Laplacian 方差 < 30.0 | 图像模糊 |

> **侧脸不被误过滤的原因**：`SUNGLASSES_EYE_PROB` 采用"双眼均低于阈值"逻辑。侧脸场景下背对侧眼睛概率约 0.10，朝向侧约 0.85，后者 > 0.70，不满足"双眼均低"条件，因此不过滤。

#### 3.1.3 处理流程

```
decodeBitmap(uri)
    → 按 MAX_IMAGE_SIZE(640) 等比缩小
    → ML Kit FaceDetector(PERFORMANCE_MODE_ACCURATE, CLASSIFICATION_MODE_ALL)
    → 质量过滤（姿态 → 面积 → 墨镜 → ）
    → cropFace(boundingBox) → 112×112 ARGB_8888
    → isBlurry(Laplacian) → TFLite Interpreter.run()
    → FloatArray[192]（L2 归一化嵌入向量）
```

#### 3.1.4 公开 API

```kotlin
// 仅返回嵌入向量（用于聚类验证/更新质心）
suspend fun extractEmbedding(uri: Uri): FloatArray?

// 返回嵌入向量 + 裁剪后的人脸 Bitmap（用于保存头像）
suspend fun extractEmbeddingWithFace(uri: Uri): Pair<FloatArray, Bitmap>?

// 从已有 Bitmap 提取（调用方负责 Bitmap 生命周期）
suspend fun extractEmbedding(bitmap: Bitmap): FloatArray?

// 释放 TFLite Interpreter 和 FaceDetector
fun close()
```

### 3.2 FaceClusterStrategy

**文件**：`photoanalysis/face/FaceClusterStrategy.kt`

实现两阶段聚类算法。

#### 3.2.1 算法参数

| 常量/参数 | 值 | 说明 |
|---|---|---|
| `threshold` | 0.65 | 贪心合并阈值：新照片与质心余弦相似度须 ≥ 此值 |
| `ANCHOR_TOLERANCE` | 0.08 | 锚点容差：新照片与锚点相似度须 ≥ threshold - 0.08 = 0.57 |
| `VERIFY_THRESHOLD` | 0.62 | 出簇验证阈值：低于此值的照片被踢出 |
| `minPhotosPerCluster` | 2 | 新簇最少照片数（已有种子簇不受限） |

#### 3.2.2 两阶段算法

**阶段一：贪心最近邻合并**

```
对每张新照片的嵌入向量 e：
  遍历所有现有簇 C：
    sim_centroid = cosine(e, C.centroid)
    sim_anchor   = cosine(e, C.anchor)
    if sim_centroid > threshold AND sim_anchor >= threshold - ANCHOR_TOLERANCE:
        记录为候选簇（取 sim_centroid 最高者）
  if 有候选簇:
      加入该簇，用加权平均更新质心
  else:
      新建簇（centroid = anchor = e）
```

**双重校验的作用**：
- `centroid`（滑动平均）：反映该簇当前的整体特征中心
- `anchor`（首张照片固定）：锁定最初定义这个簇的特征，防止质心漂移后误纳入不同人

**阶段二：出簇验证**

```
对每个簇内的每张照片（仅本批次新增照片）：
  sim = cosine(photo.embedding, cluster.finalCentroid)
  if sim < VERIFY_THRESHOLD(0.62):
      踢出该照片（丢弃，不重新分配到其他簇）
重新计算被修改簇的质心
```

#### 3.2.3 跨批次增量支持

`clusterWithCentroids()` 接受 `existingSeeds: List<ExistingClusterSeed>` 参数，将已有簇作为初始状态：

```kotlin
data class ExistingClusterSeed(
    val dbId: Int,
    val centroid: FloatArray,
    val anchor: FloatArray,
    val existingPhotoCount: Int   // 历史照片数，用于加权质心
)
```

**加权质心更新**（兼容无原始嵌入的历史照片）：

```
blended_centroid = normalize(
    historicalCentroid × historicalCount + Σ(newEmbeddings)
    ─────────────────────────────────────────────────────
                  historicalCount + newCount
)
```

**ClusterResult 中 dbId 的含义**：

| `dbId` | 含义 | Pipeline 操作 |
|---|---|---|
| `null` | 本次新建的簇 | `personDao.insert()` 写入新记录 |
| 非 `null` | 已有簇有新照片归入 | `personDao.updateCentroid()` 只更新质心，不改名称/头像 |

### 3.3 FaceClusterPipeline

**文件**：`photoanalysis/face/FaceClusterPipeline.kt`

orchestrates 完整流程，负责数据库读写。

#### 3.3.1 run() 执行步骤

```
1. 过滤 facePersonId == null 的照片（增量：已归属照片跳过）
2. personDao.getActiveForIncremental() → 解析质心/锚点为 ExistingClusterSeed 列表
3. strategy.clusterWithCentroids(unprocessed, existingSeeds)
4. 批量标记 faceScanned = 1（无论是否检测到人脸，防止重复扫描）
5. 遍历 ClusterResult：
   - dbId != null → updateCentroid + updateFacePersonId + refreshPhotoCount
   - dbId == null → insert(PersonClusterEntity) + updateFacePersonId + refreshPhotoCount
6. 保存人脸头像文件（filesDir/face_avatars/{uuid}.jpg，JPEG 85 质量）
```

#### 3.3.2 人脸头像文件

- **路径**：`{context.filesDir}/face_avatars/{UUID}.jpg`
- **大小**：约 3-5 KB（112×112 JPEG）
- **生命周期**：重新分析时 `resetAndRescanPhotos()` 删除整个目录
- **加载**：`PersonAdapter` 中判断路径以 `/` 开头则使用 `File` 对象交给 Glide 加载

## 4. 触发机制

```
App 启动 / 进入待发送页面
    └── PendingSendFragment.onResume()
            └── analysisViewModel.startFaceCluster()   ← 触发1：增量处理已识别但未扫描的照片

智能精选分析完成
    └── isLoadingLocal: true → false
            └── analysisViewModel.startFaceCluster()   ← 触发2：处理本次新识别的照片

重新分析（用户主动）
    └── analysisViewModel.resetAndRescanPhotos()
            ├── personClusterDao.deleteAll()            ← 清空人物记录
            ├── photoDao.resetFaceScanned()             ← 重置 faceScanned + facePersonId
            ├── 删除 filesDir/face_avatars/ 目录        ← 清理头像文件
            └── 重新触发完整分析流程
```

**幂等保护**：`startFaceCluster()` 检查 `_isFaceClustering.value`，聚类进行中时忽略重复调用。

**两次触发不造成人物分裂的原因**：第二次触发时 `getActiveForIncremental()` 加载第一批产生的簇作为种子，新照片优先归入已有簇。

## 5. 照片过滤条件

#### `getUnclusteredPhotos()` 查询条件

```sql
SELECT * FROM photos
WHERE isRecognized = 1    -- 已通过智能精选分析
  AND faceScanned = 0     -- 尚未执行人脸扫描
  AND isDeleted = 0       -- 未删除
  AND isPublished = 0     -- 未发布到相框
```

#### `getPhotosByFacePersonId()` 查询条件（详情页）

```sql
SELECT * FROM photos
WHERE facePersonId = :personId
  AND isDeleted = 0
  AND isPublished = 0
ORDER BY dateTaken/dateAdded DESC
```

> 聚类来源、计数统计（`refreshPhotoCount`）和详情展示三处使用相同的过滤条件，确保 `photoCount` 与实际可见照片数一致。

## 8. 性能与内存

| 指标 | 说明 |
|---|---|
| 模型加载 | 单例懒加载，`Interpreter` 和 `FaceDetector` 进程内复用 |
| Bitmap 生命周期 | `extractEmbeddingWithFace` 返回的 Bitmap 由调用方（Strategy/Pipeline）负责 `recycle()` |
| 非代表照片 Bitmap | 归入已有簇时立即 `recycle()`，仅首张保留为头像 |
| 图片解码 | 按 `MAX_IMAGE_SIZE=640` 等比缩放后再解码，避免 OOM |
| 推理并发 | 每个 `FaceClusterStrategy` 实例串行处理，`Interpreter` 设置 2 线程 |
| 头像文件 | 112×112 JPEG（85质量）约 3-5KB，存于 `filesDir` 不占相册空间 |
